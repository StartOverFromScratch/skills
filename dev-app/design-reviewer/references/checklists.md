# チェックリスト一覧

## 1. 非機能要件定義書

### 1-1. 性能・パフォーマンス
- [ ] レスポンスタイムの目標値が記載されているか（例：通常時95%tile 2秒以内）
- [ ] 同時接続数・最大スループットの想定があるか
- [ ] ピーク時の負荷を考慮しているか
- [ ] 性能目標の計測方法・基準が明示されているか

### 1-2. 可用性・信頼性
- [ ] 稼働率（SLA）の目標が記載されているか（例：99.9%）
- [ ] 計画停止・メンテナンス時間の扱いが明示されているか
- [ ] 障害時の許容ダウンタイムが定義されているか（RTO）
- [ ] データ損失の許容範囲が定義されているか（RPO）

### 1-3. セキュリティ
- [ ] 認証・認可の方式が記載されているか
- [ ] 通信の暗号化要件（HTTPS/TLS）が記載されているか
- [ ] 個人情報・機密情報の取り扱い方針があるか
- [ ] 脆弱性診断・セキュリティテストの計画があるか
- [ ] ログイン失敗時の対策（アカウントロックなど）が記載されているか

### 1-4. スケーラビリティ・拡張性
- [ ] ユーザー増加に対するスケールアップ/アウトの方針があるか
- [ ] データ増加に対する対策が記載されているか
- [ ] 将来的な機能追加を考慮した設計になっているか

### 1-5. 保守性
- [ ] ログの出力方針・保存期間が定義されているか
- [ ] デプロイ方式（手動/自動）が明示されているか
- [ ] バージョン管理・リリース管理の方針があるか

---

## 2. テスト設計書

### 2-1. テスト計画
- [ ] テストの目的・スコープが明確か
- [ ] テストフェーズ（単体・結合・システム・受入）の定義があるか
- [ ] テストの開始・終了条件（Entry/Exit Criteria）が記載されているか
- [ ] テストスケジュールが記載されているか

### 2-2. テスト観点・カバレッジ
- [ ] 機能テストの観点が網羅されているか
- [ ] 境界値・異常系のテストが含まれているか
- [ ] 非機能テスト（性能・セキュリティ）の計画があるか
- [ ] 回帰テストの範囲が定義されているか

### 2-3. テストケース
- [ ] テストケースにID・タイトルが付与されているか
- [ ] 前提条件（テスト実行前の状態）が明記されているか
- [ ] 操作手順が具体的に記載されているか
- [ ] 期待結果が明確に定義されているか
- [ ] 優先度（高/中/低）が設定されているか

### 2-4. テスト環境・データ
- [ ] テスト環境の構成が記載されているか
- [ ] テストデータの準備方針が記載されているか
- [ ] 本番データの使用有無と個人情報のマスキング方針があるか

### 2-5. 欠陥管理
- [ ] バグレポートのフォーマットが定義されているか
- [ ] 重大度・優先度の基準が定義されているか
- [ ] 修正→再テストのフローが記載されているか

---

## 3. 運用設計書

### 3-1. 監視
- [ ] 監視対象（サーバー・アプリ・DB・ネットワーク）が定義されているか
- [ ] アラートの閾値と通知先が記載されているか
- [ ] 監視ツールが明示されているか
- [ ] 死活監視（サービスが動いているかの確認）の方法があるか

### 3-2. バックアップ・リストア
- [ ] バックアップ対象・頻度・保存期間が記載されているか
- [ ] バックアップの保存場所（別リージョン等）が考慮されているか
- [ ] リストア手順・リストア時間の目標が記載されているか
- [ ] 定期的なリストア確認テストの計画があるか

### 3-3. 障害対応
- [ ] 障害レベルの定義（重大度分類）があるか
- [ ] 初動対応フローが記載されているか
- [ ] エスカレーション先・連絡先が明示されているか
- [ ] よくある障害と対処法（Runbook）が用意されているか

### 3-4. デプロイ・リリース
- [ ] デプロイ手順が記載されているか
- [ ] ロールバック手順が記載されているか
- [ ] リリース時の確認事項（デプロイ後テスト）が定義されているか
- [ ] 本番反映の承認フローが記載されているか

### 3-5. キャパシティ管理
- [ ] リソース使用量の定期確認サイクルが定義されているか
- [ ] スケールアウト・増設の判断基準が記載されているか

### 3-6. セキュリティ運用
- [ ] 定期的なパッチ適用の方針があるか
- [ ] アクセスログの取得・保存期間が定義されているか
- [ ] アカウント管理（追加・削除・棚卸し）のフローがあるか

### 4-1. 構成
- [ ] システム構成図が含まれているか
- [ ] 各コンポーネントの役割が説明されているか
- [ ] 冗長構成（単一障害点の排除）が考慮されているか

### 4-2. ネットワーク
- [ ] ネットワーク構成図・セグメント分離が記載されているか
- [ ] ファイアウォール・セキュリティグループのルールが定義されているか
- [ ] 外部接続（API・CDN・メール等）の通信要件が記載されているか

### 4-3. サーバー・インスタンス
- [ ] スペック（CPU/メモリ/ストレージ）の根拠が記載されているか
- [ ] OS・ミドルウェアのバージョンが明示されているか
- [ ] ライセンスの考慮がされているか

## 用語集（初心者向け）

| 用語 | 説明 |
|------|------|
| SLA | Service Level Agreement。サービスの稼働率などの品質目標を定めた合意 |
| RTO | Recovery Time Objective。障害発生からシステム復旧までの目標時間 |
| RPO | Recovery Point Objective。障害発生時にどこまでデータを遡って復旧できるかの目標 |
| スループット | 単位時間あたりに処理できるリクエスト数 |
| 冗長構成 | 一つのコンポーネントが壊れても動き続けられるよう予備を用意する設計 |
| Runbook | 運用作業や障害対応の手順書 |
| Entry Criteria | テスト開始の条件（例：単体テストが完了していること） |
| Exit Criteria | テスト終了の条件（例：重大バグが0件であること） |
