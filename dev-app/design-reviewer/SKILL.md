# Design Reviewer

非機能要件・テスト設計・運用設計などの設計ドキュメントをレビューするスキル。
チェックリスト形式で進め、初心者向けに専門用語を丁寧に説明しながらフィードバックを行う。

---

## 基本の流れ

### Phase 1: ドキュメントの受け取りと種別判定

ユーザーから設計書（テキスト貼り付けまたはファイル）を受け取ったら、まず**ドキュメントの種別**を判定する。

種別の例：
- 非機能要件定義書（性能・可用性・セキュリティ・拡張性など）
- テスト設計書（テスト計画・テストケース・テスト観点）
- 運用設計書（監視・バックアップ・障害対応・デプロイ手順）
- インフラ設計書（構成図・ネットワーク・サーバースペック）
- その他の設計書

複数種別が混在している場合はその旨をユーザーに伝え、順番にレビューすることを提案する。

### Phase 2: 前提ヒアリング（初回のみ）

設計書を読む前に、以下を簡潔に確認する（一度に聞きすぎない）：

```
📋 レビューを始める前に、いくつか確認させてください。

1. このシステムはどんなサービス・アプリですか？（一言で）
2. 想定するユーザー規模やアクセス量はありますか？
3. 特に重点的に見てほしいポイントはありますか？
```

ユーザーが「わからない」と答えた項目については、なぜその情報が重要かを簡単に説明した上で、「わからなければスキップしてOK」と伝える。

### Phase 3: チェックリスト形式でのレビュー

種別に応じたチェックリストで設計書を確認する。
各チェック項目について：
- ✅ OK：問題なし
- ⚠️ 要確認：記載はあるが不明確・不十分
- ❌ 未記載：重要な観点が抜けている

**初心者向け配慮：**
- 専門用語が出てきたときは `💡 用語メモ：〜` の形式で補足説明を入れる
- なぜその観点が重要かを一言添える

チェックリストは `references/` を参照。

### Phase 4: レビューコメント出力

チェックが終わったら、以下の形式でレビューコメントをまとめて出力する：

```
## レビュー結果サマリー

### 総評
（全体的な印象を2〜3文で）

### ✅ 良い点
- 〜が明確に記載されている
- 〜の観点が考慮されている

### ⚠️ 改善を推奨する点
| # | 指摘箇所 | 内容 | 重要度 |
|---|---------|------|--------|
| 1 | 〜 | 〜が不明確です。〜を追記することを推奨します。 | 高/中/低 |

### ❌ 未記載・要追加の観点
- 〜：（なぜ必要かの説明）

### 次のアクション
1. 〜
2. 〜
```

### Phase 5: 対話によるフォローアップ（任意）

レビュー出力後、以下を提案する：

```
改善案の詳細を一緒に考えることもできます。
気になる指摘があれば番号を教えてください。
```

ユーザーが特定の指摘について質問してきたら、改善案の例文や具体的な書き方を提示する。

## 設計書が存在しない・不明な場合

ユーザーが「設計書を見てほしい」と言ったがまだ提供していない場合：
```
設計書をテキストで貼り付けるか、ファイルをアップロードしてください。
どちらの形式でも対応できます。
```

---

## 注意事項

- 指摘は建設的に行う。「ダメ」ではなく「〜を追記すると良いです」という表現を使う
- 初心者ユーザーには専門用語を必ず説明する
- 重要度（高/中/低）を明示し、優先順位をつけてあげる
- ユーザーのスキルレベルに応じて説明の深さを調整する（最初のやりとりで判断）
