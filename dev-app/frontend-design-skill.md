# フロントエンド設計スキル

## 概要

このスキルは、フロントエンド設計を対話形式で進め、設計書（Markdown or docx）として出力するためのものです。
採用技術はユーザーが決定するまで特定せず、確定した技術に合わせて設計内容を具体化します。

---

## ステップ0：前提確認

設計を始める前に、以下を確認してください。

### 確認すること

1. **認識合わせ**：「フロントエンド設計を進めます。まず確認させてください」と一言添える
2. **技術スタックの確定**（未決定の場合は選択肢を提示して確認する）
3. **スコープの確定**（何を設計書に含めるか）
4. **要件定義書・バックエンド設計書の参照**（ある場合は内容を把握する）

### 技術選定の確認項目

| 項目 | 確認内容 |
|------|---------|
| フレームワーク | React / Vue / Svelte / Angular など。未決定なら要件に応じて比較説明する |
| UIライブラリ | フレームワークに対応するものを候補提示。未決定なら比較説明する |
| 状態管理 | フレームワークに対応するものを候補提示（Redux / Zustand / Pinia / Svelte store など） |
| API通信 | fetch / axios / React Query / SWR など |
| 出力形式 | Markdown / docx |

### 技術選定の比較説明方針

技術が未決定の場合は、以下の観点で比較説明してユーザーの判断を引き出す：

- **規模感**：小規模個人 / 中規模チーム / 大規模エンタープライズ
- **学習コスト**：既存知識との親和性
- **エコシステム**：ライブラリの豊富さ・コミュニティの活発さ
- **要件との適合**：画面の複雑さ・リアルタイム性・パフォーマンス要件

比較説明後は必ずユーザーに選択を委ね、決定した技術を明示的に確認してから次に進む。

### スコープ確認

以下の項目から今回含めるものを確認する：

- ディレクトリ構成（ほぼ必須）
- API通信層の設計（重要度：高）
- 状態管理方針（重要度：高）
- 画面設計・ワイヤーフレーム（重要度：高）
- エラーハンドリング方針（中）

## ステップ1：ディレクトリ構成

採用フレームワークの規約に従い、以下の層分離を原則とする。

```
画面層（pages/views/routes）
  → ロジック層（hooks/composables/services）
    → API通信層（repositories/api）
      → バックエンド
```

### ディレクトリ構成の設計方針

フレームワークに関わらず、以下の役割分離を意識して具体化する：

| ディレクトリ | 役割 | フレームワーク別の名称例 |
|------------|------|----------------------|
| 画面ルーティング | URLと画面の対応 | pages（Next/Nuxt）/ routes（React Router / Vue Router） |
| UIコンポーネント | 表示・操作の単位 | components/ |
| ビジネスロジック | 状態操作・副作用 | hooks/（React）/ composables/（Vue）/ services/ |
| 状態管理 | グローバル状態 | store/（Redux）/ stores/（Pinia）/ context/ |
| API通信 | HTTP通信のみ | repositories/ / api/ / services/api/ |
| 型定義 | TypeScript型 | types/ |
| ユーティリティ | 共通関数 | utils/ / lib/ |

確定した技術スタックに合わせてディレクトリ名・ファイル名を具体化する。

## ステップ2：API通信層の設計

### 設計原則（技術非依存）

- **通信層の分離**：API通信のロジックを画面・ビジネスロジックから切り離す
- **エラーの移譲**：通信層はエラーをそのまま投げ上げ、上位層でハンドリングする
- **baseURLの外部化**：環境変数（`.env`）で管理し、コードにハードコードしない

### 各フレームワークでの実装パターン

**共通パターン（fetch / axios）：**
```typescript
// repositories/xxxRepository.ts
const BASE_URL = import.meta.env.VITE_API_BASE  // または process.env.NEXT_PUBLIC_API_BASE など

export const xxxRepository = {
  async fetchAll(params?: QueryParams): Promise<Xxx[]> {
    const res = await fetch(`${BASE_URL}/endpoint`, { /* params */ })
    if (!res.ok) throw new Error(res.statusText)
    return res.json()
  },
  async create(body: CreateRequest): Promise<Xxx> {
    const res = await fetch(`${BASE_URL}/endpoint`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(body)
    })
    if (!res.ok) throw new Error(res.statusText)
    return res.json()
  },
}
```

確定した技術に合わせて、`$fetch`（Nuxt）/ `axios`（汎用）/ `ky`（汎用）などに読み替える。

### エラーハンドリング方針

ビジネスロジック層でcatchし、UIに適した形で通知する。通知手段はUIライブラリに依存する（Toast / Alert / モーダルなど）。

| ステータスコード | ケース | 表示方針 |
|----------------|--------|---------|
| 400 | バリデーションエラー | 操作できない理由を明示 |
| 404 | 存在しないリソース | 対象が見つかりませんでした |
| 422 | リクエストエラー | 入力内容を確認してください |
| 500 | サーバーエラー | エラーが発生しました。しばらく後に再試行してください |

```typescript
// ビジネスロジック層でのエラーハンドリング例（擬似コード）
async function doAction(id: number) {
  try {
    const result = await xxxRepository.doSomething(id)
    store.upsert(result)
  } catch (e: any) {
    if (e.statusCode === 400) {
      notify({ type: 'error', message: 'できません：理由を明示' })
    } else {
      notify({ type: 'error', message: 'エラーが発生しました' })
    }
  }
}
```

## ステップ3：状態管理方針

### 分割の基本原則

- 概念的に独立しているものは別の状態単位に分ける
- 文脈が異なる操作は独立した状態として管理し、意図を明示する
- グローバル状態に載せるのは「複数の画面をまたいで参照・更新されるもの」に限定する

### フレームワーク別の状態管理パターン

| フレームワーク | 状態管理ライブラリ | ストア/スライスの単位 |
|-------------|----------------|-------------------|
| React | Redux Toolkit / Zustand / Jotai | slice / store（機能ごと） |
| Vue 3 | Pinia | store（機能ごと） |
| Svelte | svelte/store | writable / derived（機能ごと） |
| Angular | NgRx / Signals | feature store |

### 状態管理の設計で確認すること

- どのデータをグローバル状態に載せるか（vs ローカルstate）
- 状態の単位をどの粒度で分けるか（機能ごと / リソースごと）
- 楽観的更新（Optimistic Update）を採用するか
- キャッシュ戦略（React Query / SWR などのサーバー状態管理を使うか）

## ステップ4：画面設計（ワイヤーレベル）

### 設計の進め方

1. 画面一覧をテーブルで整理（画面名・パス・役割）
2. 各画面をASCIIワイヤーフレームで表現
3. 各画面の主要コンポーネントを記載

### ワイヤーフレームの書き方

```
┌─────────────────────────────────────────┐
│  画面タイトル          [アクションボタン] │
│  ─────────────────────────────────────  │
│  [フィルタ・タブなど]                    │
│  ─────────────────────────────────────  │
│  ┌─────────────────────────────────┐   │
│  │ コンテンツエリア                 │   │
│  │ 主要情報・操作ボタン             │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

### 画面設計で確認すること

- 各画面の主目的は何か
- バナー・アラートなど横断的な通知をどこに置くか
- フォームの必須項目・任意項目・条件付き表示フィールド
- 一覧→詳細→編集の遷移フロー

## ステップ5：設計書の出力

### 出力形式

ユーザーの指定に従い以下のいずれかで出力する：

- **Markdown**（`.md`）：デフォルト推奨。GitやNotionへの貼り付けに便利
- **Word文書**（`.docx`）：docxスキルを参照して作成

### Markdownの構成テンプレート

```markdown
# [システム名] フロントエンド設計書

作成日: YYYY-MM-DD | バージョン: X.X

---

## 0. 構成概要（採用技術スタック一覧）
## 1. ディレクトリ構成
## 2. API通信層の設計
## 3. 状態管理方針
## 4. 画面設計（ワイヤーレベル）
## 5. 次回スコープ
```

### 引き継ぎサマリーの生成

設計書とあわせて、次のセッションで使える引き継ぎサマリーをMarkdownで出力することを推奨する。含める内容：

- 確定した技術スタックと選定理由
- 今回の設計決定と理由
- 次回やること

## 設計思想メモ

- **層の分離を徹底する**：画面層はロジック層しか呼ばない。通信はAPI層に閉じ込める
- **文脈の異なる概念は別の状態単位へ**：機能が異なるものを同じストアに詰め込まない
- **エラーハンドリングはロジック層に集める**：通信層では投げ上げるだけにする
- **環境変数でbaseURLを管理する**：コードにURLをハードコードしない
- **条件付き表示はコンポーネント内で完結させる**：親から渡すフラグで制御する
