# バックエンド設計スキル

## 概要

要件定義書をもとに、FastAPI + SQLite構成のバックエンド設計を段階的に完成させる。
各ステップで設計上の選択肢を提示し、設計思想を交えた比較でユーザーの判断を引き出す。
確定した決定事項を蓄積しながら進み、最終的に設計書（Markdown）と引き継ぎサマリーを出力する。

---

## 進め方の原則

### 1. 最初に認識合わせをする

作業開始前に以下を確認する。

- 今日やりたい設計作業はどれか
- 全体を俯瞰してから個別に進むか、特定の箇所から始めるか

### 2. 各ステップで設計思想を交えて比較する

選択肢が複数ある場合は以下の形式で比較する。

```
案A：XXX
  思想：〜という考え方に基づく
  メリット：〜
  トレードオフ：〜

案B：YYY
  思想：〜という考え方に基づく
  メリット：〜
  トレードオフ：〜

→ 推奨：〜（理由：〜）
```

ユーザーから「どう違うか」「なぜそうするか」という質問が来た場合は、設計思想レベルまで掘り下げて説明する。具体例は過去の設計を参照するのではなく、その場のシステム要件に即して生成する。

### 3. 確定事項を蓄積する

各ステップの終わりに確定事項をまとめて表示する。

```
確定事項：
・完了したステップ名：完了
・選択した設計方針：確定（理由：〜）
...
```

### 4. 確認を省略しない

設計の各ポイントで必ずユーザーに確認を取る。推測で進めない。

## 設計ステップ

### Step 1: DFD（データフロー図）

#### レベル0（コンテキスト図）
システム全体を1プロセスとして表現する。外部エンティティ・データの入出力・DBを明示する。

#### レベル1（主要プロセス分解）
システム内部を主要プロセスに分解する。各プロセスに番号を付与し、対応するAPIエンドポイントとの対応を示す。

**確認観点：**
- スコープ外の機能が正しく除外されているか
- プロセスの粒度が適切か
- 外部エンティティの数・種類が要件と一致しているか

---

### Step 2: API一覧

DFDのプロセスをAPIリソースに対応させる。

**設計方針の選択観点：**
- 動的計算状態（DBのstatusカラムに持たない状態）をどう扱うか
- 文脈が通常のCRUDと異なる操作を独立ルーターにするか統合するか
- パス方式とクエリパラメータ方式のどちらが設計思想と整合するか

---

### Step 3: 状態遷移とAPI対応

statusの全値を列挙し、各遷移がどのエンドポイントで発生するかを対応表で整理する。

**確認観点：**
- 各statusへの遷移を引き起こすエンドポイントが明確か
- 動的計算で判定する状態とDBに持つ状態が区別されているか
- 特定の操作（完了・削除など）を専用エンドポイント経由に限定する必要があるか
- 将来の実装変更が発生したときのAPIへの影響範囲

---

### Step 4: APIスキーマ詳細

各エンドポイントのリクエスト・レスポンスフィールドを定義する。

**確認観点：**
- 操作の成否バリデーションをどこで行うか（DB・サービス層・APIレイヤー）
- エラー時の挙動（拒否 vs 警告）をシステムの設計思想と整合させているか
- リソース間の参照関係（切り出し・親子・逆参照）が必要かどうか
- 参照関係が必要な場合のトレーサビリティ確保方法

---

### Step 5: ディレクトリ構成

FastAPIの4層構造を基本とする。

```
backend/
├── main.py
├── alembic.ini
├── alembic/
│   ├── env.py
│   └── versions/
├── app/
│   ├── api/          # ルーター（エンドポイント定義）
│   ├── models/       # SQLAlchemyモデル
│   ├── schemas/      # Pydanticスキーマ
│   ├── services/     # ビジネスロジック
│   ├── db/           # セッション管理
│   └── core/         # 設定値
└── tests/
```

**確認観点：**
- 文脈が異なる操作をディレクトリ構造として明示するか
- ビジネスロジックをservices層に集約できているか
- 将来の機能追加時にどのレイヤーに追加するかが明確か

---

### Step 6: マイグレーション設計

#### ツール選定観点
- 将来のRDB移行が要件にあるか
- スキーマ変更の履歴管理が必要か
- 開発速度とマイグレーション管理の複雑さのバランス

#### SQLite固有の制約への対処
`render_as_batch=True`を初期設定することで、カラム削除・変更・外部キー変更に対応する。

#### テーブル間の参照関係の確認観点
- 循環参照が発生しないか
- 循環参照がある場合、外部キー制約を維持するか否か
  - 維持する場合：DBレベルの整合性保証、削除処理の順序管理が必要
  - 維持しない場合：アプリ層での整合性保証、削除処理がシンプル
- `ondelete`の設定（CASCADE / SET NULL / RESTRICT）が適切か

#### インデックス設計観点
- クエリ頻度が高いカラムはどれか
- 動的判定に使うカラムにインデックスがあるか

---

### Step 7: OpenAPI定義（yaml）

バックエンド設計の締めとして、全エンドポイントのOpenAPI定義をyaml形式で出力する。

構成：
- `components/schemas`：共通型定義（Enum・Request・Response）
- `components/responses`：共通エラーレスポンス
- `paths`：全エンドポイント定義

## 最終出力

設計が完了したら以下の2ファイルを出力する。

### 設計書（backend_design.md）

以下のセクションを含む。

1. DFD（レベル0・レベル1・プロセスとAPIの対応）
2. API一覧（設計方針・プロセスごとのエンドポイント表）
3. 状態遷移とAPI対応（遷移図・対応表・設計方針の根拠）
4. APIスキーマ詳細（共通型定義・エンドポイントごとのRequest/Response）
5. ディレクトリ構成（ツリー・4層分離の設計意図）
6. マイグレーション設計（設定・モデル定義・ファイル構成・インデックス・SQLite制約対処）
7. OpenAPI定義（yaml）
8. 今後の課題（未着手領域の明示）

### 引き継ぎサマリー（handover_summary.md）

以下のセクションを含む。

- 今日完了したこと
- 確定した設計決定と理由（なぜその選択をしたかを記録）
- データモデルの変更点（要件定義書からの差分）
- 次回やること（優先度付き）
- 参照ドキュメント
